# PHP: Symfony Demo meets Modular, Microservice-ready Architecture - Part 2

Источник: https://www.kode-krunch.com/2021/12/php-symfony-modular-architecture-demo-part-2.html

Дата публикации: 2021-12-01

Дата обращения: 2026-04-28

## Ключевые идеи

- Эта часть переводит идеи из `Part 1` в конкретную структуру кода Symfony 6 demo-приложения и показывает, какие правила помогают потом вынести модуль в отдельный сервис без смены архитектурной модели.
- В общем Infrastructure-слое автор выделяет две критические абстракции: транзакции и события. Для транзакций он добавляет собственный слой с `afterCommit` и `afterRollback`, чтобы бизнес-код знал точный момент, когда изменения уже зафиксированы и можно безопасно публиковать события.
- Дополнительно вводится паттерн "Entity Manager per transaction": каждая транзакция получает свой свежий `Entity Manager`, который очищается и закрывается по завершении работы. Это делает поведение CLI-обработки и повторных транзакций более предсказуемым.
- Symfony Messenger используется не напрямую, а через тонкую событийную абстракцию. Она прячет транспорт, собирает общую логику логирования и обработки ошибок в одном месте и сохраняет одинаковую модель работы для sync и async доставки.
- Каждый модуль имеет одну и ту же внутреннюю форму: `API`, `Domain`, `Persistence`. Такая повторяемость нужна не ради "идеальных папок", а чтобы разработчики быстро перемещались между модулями и не переучивали проект заново.
- `API`-слой задаёт допустимые способы общения с модулем: `Command` для изменений, `Query` для чтения, `Inbound Event` для входящих сообщений, плюс контроллеры и `Module API Interface` как единая точка входа.
- `Domain`-слой содержит собственно бизнес-логику: DTO, исходящие события, классы логики, интерфейсы репозиториев и транзакций. Он работает через абстракции и не должен зависеть от деталей Doctrine или Symfony.
- `Persistence`-слой реализует репозитории и замыкает в себе технологические детали. Важно, что сущности Doctrine не должны "протекать" выше этого слоя.
- Главная тестовая мысль статьи: unit tests должны проверять не отдельный класс, а модульное поведение. Классы - это деталь реализации, а требования бизнеса живут на более широком уровне.
- Поэтому unit tests идут через `Module API Interface`, используют in-memory репозитории и обходятся без I/O и Symfony-инфраструктуры. Integration tests покрывают реальные HTTP round-trip и базу данных, а совместимость событий между модулями проверяется через consumer-driven contracts на JSON-контрактах.
- В финале автор трактует стандартизацию архитектуры не как ограничение творчества, а как снятие рутинных решений: разработчики меньше думают о раскладке классов и соглашениях и больше - о предметной логике.

## Что забрать в permanent notes

- События безопаснее публиковать из явного post-commit механизма, а не из кода, который ещё находится внутри незавершённой транзакции.
- Для модульного монолита полезно зафиксировать один официальный вход в модуль через его API и не давать framework/persistence деталям просачиваться в доменный слой.
- Быстрые модульные unit tests через API модуля часто лучше защищают от регрессий, чем большая масса тестов, жёстко привязанных к классам.

## Цитаты

- `Interaction with the Module should always be done on the API Level.`
- `Technology-specific implementation should never leak into the higher Layers.`
- `the appropriate scope for any unit test is a Module scope`
