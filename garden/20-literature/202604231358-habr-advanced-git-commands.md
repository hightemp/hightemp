# Вы больше не новичок в Git. Повышаем квалификацию с помощью 7 команд

Источник: https://habr.com/ru/companies/ruvds/articles/978320/

Дата обращения: 2026-04-23

## Ключевые идеи

- Переход от базового Git к уверенному ежедневному использованию начинается не с редких сложных команд, а с улучшения привычного workflow: алиасы, отдельные рабочие деревья, аккуратное редактирование локальных коммитов и навигация по истории.
- Алиасы полезны не только как сокращения для длинных команд, но и как способ зафиксировать собственный словарь операций: `unstage`, быстрый компактный `status`, удобный `log`, частые варианты `commit --amend`.
- `git worktree` решает задачу параллельной работы с несколькими ветками без постоянного `stash`/`checkout`: hotfix, ревью или эксперимент можно открыть в отдельной директории того же репозитория.
- `git commit --amend` подходит для исправления последнего локального коммита: добавить забытый файл, поправить мелочь или изменить сообщение. Его нельзя применять бездумно к уже опубликованной истории.
- `git cherry-pick` переносит конкретный коммит между ветками без merge всей ветки. Это удобно для точечного багфикса, но конфликты остаются обычной частью процесса.
- `git reflog` нужно помнить как локальный журнал перемещений `HEAD`: он помогает восстановиться после ошибочного `reset`, неудачного `rebase` или потерянного локального коммита.
- `git rebase -i` превращает локальную историю в редактируемый сценарий: можно переименовывать, склеивать, переставлять или удалять коммиты. Главный риск - переписать уже опубликованную историю.
- `git log --graph` и `tig` помогают видеть форму истории, а не только список коммитов; это снижает риск механически делать merge/rebase без понимания текущего графа.
- При переносе примеров из статьи в личную шпаргалку стоит сверять синтаксис с официальной документацией: например, сообщение для `commit --amend` задается через `-m`, а не `--m`.

## Что забрать в permanent notes

- `git worktree` полезен как альтернатива `stash` при параллельных задачах: он разделяет рабочие директории, но оставляет общий репозиторий и историю.
- Команды, переписывающие историю (`amend`, `rebase -i`, `reset`), безопасны в первую очередь для локальной, не опубликованной работы.
- `git reflog` - аварийный журнал локальных перемещений `HEAD`, который стоит проверять до паники после потери коммитов.
- Хорошая Git-практика - сначала понимать граф истории, а уже потом выбирать между merge, rebase, cherry-pick и reset.

## Цитаты

- `git worktree add <path> <branch>`
- `git commit --amend --no-edit`
- `git cherry-pick <commit>`
- `git reflog`
- `git rebase -i HEAD~3`
- `git log --graph`
