# RFC 3484 on Linux

Источник: https://www.akkadia.org/drepper/linux-rfc3484.html

Автор: Ulrich Drepper, 2007-09-04.

## Ключевые идеи

- RFC 3484 описывает два связанных решения при открытии соединения: какой адрес назначения выбрать, если у целевого имени несколько IPv4/IPv6-адресов, и какой локальный адрес источника использовать, если у машины несколько адресов или интерфейсов.
- В Linux эта логика распределена между ядром и user space: ядро выбирает адрес источника при `connect()`/`sendto()`, а glibc `getaddrinfo()` сортирует адреса назначения с учетом пар "источник - назначение".
- Порядок, который возвращает `getaddrinfo()`, является частью сетевой политики системы. Приложению лучше пробовать адреса в этом порядке, а не писать собственную сортировку.
- Drepper отдельно подчеркивает `AI_ADDRCONFIG`: без него приложение может получить IPv6-адреса даже на машине, где реального IPv6-подключения нет, и начать с заведомо плохой попытки соединения.
- Главная практическая проблема RFC 3484 в статье: старые правила могли предпочитать глобальный IPv6-адрес, даже когда у клиента есть только site-local/ULA IPv6 и рабочий private IPv4 через NAT. В таком случае IPv4 реально подключается, а IPv6 нет.
- glibc обходит эту проблему через отдельные `label` для global IPv6, site-local IPv6, ULA и IPv4-mapped адресов. Так сортировка может предпочесть глобальный IPv4 перед глобальным IPv6, если локальный IPv6 не подходит.
- `/etc/gai.conf` позволяет администратору заменить таблицы `label` и `precedence`, которые glibc использует для сортировки. Если файла нет, glibc использует дефолтную политику.

## Контекст по времени

- Статья написана в 2007 году и говорит, что RFC 3484 на тот момент не был заменен. Сейчас RFC 3484 надо читать как исторический и implementation-oriented источник; современная версия правил выбора адресов описана в RFC 6724.
- Несмотря на возраст, статья полезна тем, что объясняет Linux-реализацию: почему часть поведения находится в ядре, часть в glibc, а `/etc/gai.conf` влияет не на весь выбор адресов, а на таблицы политики в `getaddrinfo()`.

## Практические детали

- `gethostbyname()` и другие старые интерфейсы не получают RFC 3484-сортировку; для современного поведения нужен `getaddrinfo()`.
- Для оценки адресов `getaddrinfo()` должен узнать подходящий адрес источника для каждого кандидата назначения. Поэтому поведение зависит от текущих интерфейсов, маршрутов и политики ядра, а не только от DNS-ответа.
- `nscd` не решает эту часть задачи: кешировать можно сырые данные, но итоговая сортировка должна учитывать текущее состояние машины.
- `/usr/share/docs/glibc-common-*/gai.conf` в статье упоминается как пример конфигурации, а `/etc/gai.conf` - как место для локальной политики.

## Что забрать в permanent notes

- DNS-ответ и порядок подключения - разные уровни: DNS возвращает кандидатов, а `getaddrinfo()` и ядро выбирают практически пригодный порядок.
- Совместимость с системным сетевым поведением включает glibc, ядро, routing state и локальную policy-конфигурацию, а не только RFC.
- `/etc/gai.conf` - не универсальный переключатель IPv4/IPv6, а способ менять таблицы адресной политики для `getaddrinfo()`.
