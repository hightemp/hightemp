# PHP: Symfony Demo meets Modular, Microservice-ready Architecture - Part 1

Источник: https://www.kode-krunch.com/2021/11/php-symfony-modular-architecture-demo-part-1.html

Дата публикации: 2021-12-01

Дата обращения: 2026-04-27

## Ключевые идеи

- Автор критикует официальный Symfony Demo как слишком маленький и слишком "учебный" пример для разговоров о large-scale приложениях: там нет модульности, почти нет явного разделения слоёв, не разведены unit и integration tests, а безопасность построена на stateful session-подходе.
- Главная мысль статьи: масштабируемость приложения - это не только производительность под нагрузкой, но и способность команды разработки расти без взаимных блокировок и хаоса в общей кодовой базе.
- Модульность здесь нужна прежде всего для масштабирования разработки. Микросервисы рассматриваются как модули, которые можно независимо деплоить; значит, хорошая архитектура должна начинаться как modular monolith, из которого потом можно выделять сервисы без перелома модели.
- Из этого следуют строгие правила для модулей: у каждого свой процесс и своя база данных, прямой доступ к данным другого модуля запрещён, межмодульная интеграция должна идти через высокоуровневые API или события, а безопасность лучше строить stateless-способом или через API gateway.
- Наивная синхронная интеграция между модулями быстро создаёт сетевой шум и хрупкость: таймауты, ретраи, circuit breakers, программные join между наборами данных и в худшем случае cascading failures.
- Автор предлагает другой подход: каждый модуль хранит локально те данные, которые нужны ему для работы, в виде read model, и обновляет их событиями от процесс-владельца данных.
- Ключевая смена мышления: в distributed system не надо заставлять систему вести себя синхронно. Если модуль каждый раз в реальном времени спрашивает другой модуль, он разрушает асинхронную природу системы и делает себя зависимым от её сбоев.
- Read models в статье противопоставлены классическому cache: они являются явной частью архитектуры и data access layer, обновляются по событиям, хранятся как часть модели модуля и поддерживаются как обычный код, а не как "невидимый ускоритель".
- Вместо data ownership автор предлагает мыслить process ownership: модуль отвечает не просто за "свои таблицы", а за свой процесс и вправе держать локальные копии внешних данных в удобной для себя форме.
- Цена подхода тоже названа прямо: больше начального кода и инфраструктуры, eventual consistency вместо привычных ACID-транзакций между частями системы, необходимость baseline-событий и отдельного подхода к проверке контрактов между модулями, например через consumer-driven contracts.

## Что забрать в permanent notes

- Modular monolith полезен как стартовая форма архитектуры, если цель - сохранить границы модулей до реальной необходимости выносить их в отдельные сервисы.
- Синхронные запросы между модулями в runtime почти всегда покупают удобство ценой хрупкости; локальные read models и события уменьшают связанность лучше, чем распределённый cache и запросы "по месту".
- Для большой системы важно масштабировать не только железо и throughput, но и команду разработки; модульность решает обе задачи, но разными механизмами.

## Цитаты

- `Micro-services are Modules that can be deployed independently`
- `A Distributed System is Asynchronous by Nature`
- `Every Module should have all the data it needs to operate stored locally, in a Read Model`
