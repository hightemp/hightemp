# Напиши что должен помнить сеньер про MongoDB при работе с nodejs

## 1. Не думать о MongoDB как о “просто JSON в базе”
MongoDB — это не “склад объектов”, а полноценная БД со своими ограничениями и особенностями:

- есть стоимость `$lookup`, `$unwind`, `$group`
- есть влияние структуры документа на производительность
- есть ограничения на размер документа: **16 MB**
- есть важность правильных индексов
- есть разница между удобной схемой записи и удобной схемой чтения

## 2. Схема нужна, даже если база “schemaless”
Сеньор должен помнить:

- **schemaless != schema-free**
- схема должна быть явно определена на уровне приложения
- нужно валидировать данные:
  - через `mongoose schema`
  - или через `zod` / `joi`
  - или через MongoDB schema validation

Если схему не контролировать, коллекция быстро превращается в набор несовместимых документов.

## 3. Правильно выбирать: embedding vs referencing
Главный вопрос моделирования:

- **Embedding** — когда данные живут вместе и читаются вместе
- **Referencing** — когда данные большие, переиспользуются или часто изменяются независимо

Обычно стоит помнить:

- маленькие и зависимые сущности — встраивать
- большие, растущие, независимо живущие — ссылать
- массивы, которые могут бесконечно расти, нельзя бездумно встраивать в документ

Пример плохой идеи:

```js
{
  _id: userId,
  comments: [ /* сотни тысяч комментариев */ ]
}
```

Это приводит к:
- росту документа
- проблемам с обновлением
- риску упереться в лимит размера документа
- ухудшению производительности

## 4. Индексы — это не “потом добавим”
Сеньор должен проектировать запросы вместе с индексами.

Нужно помнить:

- индекс должен соответствовать реальным query patterns
- каждый индекс ускоряет чтение, но замедляет запись
- лишние индексы увеличивают память и storage
- порядок полей в compound index критичен
- `sort` без подходящего индекса может быть очень дорогим
- `explain()` надо использовать регулярно, а не только когда “всё уже тормозит”

Пример:

```js
db.orders.find({
  customerId: "...",
  status: "paid"
}).sort({ createdAt: -1 })
```

Для такого кейса полезен индекс вроде:

```js
{ customerId: 1, status: 1, createdAt: -1 }
```

## 5. Всегда проверять запросы через `explain`
В Node.js-коде многие пишут запрос и считают, что раз он работает — значит он нормальный. Это ошибка.

Нужно смотреть:

- используется ли `IXSCAN`, а не `COLLSCAN`
- сколько документов просмотрено
- сколько документов реально возвращено
- нет ли дорогих стадий в aggregation pipeline

Если запрос читает 100000 документов ради 20 результатов — это уже проблема.

## 6. Осторожно с aggregation pipeline
Aggregation — мощный инструмент, но дорогой.

Важно помнить:

- чем раньше `$match`, тем лучше
- `$project` полезен для уменьшения объёма данных
- `$lookup` — не бесплатный join
- `$unwind` может взрывать количество документов
- `$group` может жрать память
- пайплайн надо проектировать как поток с ранней фильтрацией

Хороший принцип:
- сначала отфильтровать
- потом сократить поля
- потом join/group
- потом финальная сортировка/форматирование

## 7. Не тащить лишние поля в Node.js
Если нужен только кусок документа — выбирай только его.

### Для native driver:
```js
collection.find(
  { status: 'active' },
  { projection: { _id: 1, email: 1, name: 1 } }
)
```

### Для Mongoose:
```js
User.find({ status: 'active' }).select('_id email name')
```

Это уменьшает:
- сетевой трафик
- расход памяти в Node.js
- стоимость сериализации/deserialization

## 8. `skip/limit` плохо масштабируется
Для больших пагинаций `skip` становится дорогим.

Плохо:

```js
db.posts.find().sort({ createdAt: -1 }).skip(20000).limit(20)
```

Лучше использовать **cursor-based pagination**:

```js
db.posts.find({
  createdAt: { $lt: lastSeenCreatedAt }
}).sort({ createdAt: -1 }).limit(20)
```

А ещё лучше — устойчивый курсор по нескольким полям, например:
- `createdAt`
- `_id`

## 9. Нужно помнить про read/write concern
Сеньор должен осознанно выбирать консистентность и durability, а не жить на дефолтах “как-нибудь”.

Важно понимать:

- `readConcern`
- `writeConcern`
- `readPreference`

Например:
- что читать с secondary может быть ок для аналитики
- но опасно для критичных user-facing сценариев
- что подтверждение записи влияет на надёжность и latency

## 10. Транзакции — не замена хорошему моделированию
MongoDB поддерживает транзакции, но:

- они тяжелее обычных операций
- повышают latency
- не должны использоваться как костыль вместо нормальной денормализации и атомарных обновлений документа

Если можно решить задачу одной атомарной операцией над документом — это обычно лучше транзакции.

## 11. Пользоваться атомарными операторами, а не read-modify-write
Типичная ошибка в Node.js:

```js
const user = await users.findOne({ _id: userId })
user.balance += 100
await users.updateOne({ _id: userId }, { $set: { balance: user.balance } })
```

Это создаёт race condition.

Правильно:

```js
await users.updateOne(
  { _id: userId },
  { $inc: { balance: 100 } }
)
```

Часто нужны:
- `$inc`
- `$set`
- `$unset`
- `$push`
- `$addToSet`
- `$pull`
- positional operators
- array filters

## 12. Идемпотентность важнее, чем кажется
При работе с очередями, ретраями, внешними API сеньор должен проектировать операции так, чтобы повтор не ломал данные.

Например:
- использовать уникальные ключи
- делать upsert
- хранить `operationId`
- ставить unique index на бизнес-идентификаторы

## 13. `upsert` — полезно, но опасно
`upsert` удобно, но надо помнить:

- фильтр должен быть действительно уникальным
- иначе можно случайно создать неожиданные документы
- желательно опираться на unique indexes
- важно понимать, какие поля попадут в созданный документ

Пример:

```js
await users.updateOne(
  { email },
  {
    $set: { lastLoginAt: new Date() },
    $setOnInsert: { createdAt: new Date() }
  },
  { upsert: true }
)
```

## 14. Следить за connection pool в Node.js
Для Node.js это особенно важно.

Нужно помнить:

- не создавать новое подключение на каждый запрос
- использовать один общий `MongoClient`
- настраивать pool size осознанно
- учитывать burst traffic
- следить за таймаутами:
  - `serverSelectionTimeoutMS`
  - `socketTimeoutMS`
  - `maxPoolSize`
  - `minPoolSize`

Плохая практика:

```js
app.get('/users', async (req, res) => {
  const client = await MongoClient.connect(uri)
  const users = await client.db().collection('users').find().toArray()
  await client.close()
  res.json(users)
})
```

## 15. Не блокировать Node.js огромными выборками
Ошибка — делать огромный `toArray()`.

Плохо:

```js
const docs = await collection.find({}).toArray()
```

Если документов много, это:
- съест память
- увеличит GC pressure
- может положить процесс

Лучше:
- стримить курсор
- обрабатывать батчами
- использовать лимиты
- делать backpressure-aware обработку

Пример:

```js
const cursor = collection.find({})

for await (const doc of cursor) {
  // обработка
}
```

## 16. Mongoose — это удобно, но это не MongoDB
Сеньор должен помнить разницу между:
- возможностями MongoDB
- абстракцией Mongoose
- накладными расходами Mongoose

Важно понимать:
- когда нужен native driver
- когда `lean()` обязателен
- когда populate превращается в проблему
- когда middleware Mongoose усложняет предсказуемость

Если не нужна document-модель и магия ORM-like слоя, native driver часто проще и быстрее.

## 17. В Mongoose часто нужен `lean()`
Если не нужны методы документа, virtuals и прочая обвязка:

```js
const users = await User.find({ active: true }).lean()
```

Это уменьшает overhead на гидрацию документов и экономит память/CPU.

Особенно важно в:
- списках
- API-ответах
- фоновых джобах
- high-load ручках

## 18. `populate()` не бесплатен
`populate()` удобен, но сеньор должен понимать:

- сколько запросов реально делается
- можно ли заменить на denormalization
- можно ли заменить на aggregation `$lookup`
- не тащатся ли лишние поля
- не возникает ли N+1-подобное поведение

## 19. TTL, partial, sparse, unique indexes — это must know
Нужно хорошо знать специальные индексы:

- **TTL index** — для временных данных, сессий, токенов, логов
- **partial index** — индексировать только часть документов
- **unique index** — гарантировать бизнес-инварианты
- **text index** — применять осознанно, не как замену полноценному search
- **sparse index** — понимать аккуратно, чаще сейчас полезнее partial index

Пример:
- уникальный email только для не удалённых пользователей можно решать через partial unique index

## 20. Soft delete влияет на всё
Если в системе есть `deletedAt` или `isDeleted`, это должно учитываться везде:

- во всех запросах
- в индексах
- в unique constraints
- в агрегациях
- в бизнес-логике восстановления

Иначе потом получаются:
- “пропавшие” записи
- дубликаты email/slug
- неожиданные конфликты

## 21. Времена и даты хранить правильно
Помнить:

- хранить даты в UTC
- не хранить бизнес-время как строку
- разделять:
  - момент времени
  - локальную timezone-логику
- аккуратно работать с диапазонами дат и индексами

В Node.js легко случайно намешать:
- `Date`
- ISO-строки
- локальное время сервера
- timezone пользователя

## 22. ObjectId — это не просто строка
Сеньор должен помнить:

- `ObjectId` содержит временную компоненту
- сортировка по `_id` часто коррелирует со временем создания, но не всегда это корректная замена `createdAt`
- сравнение строки и `ObjectId` может ломать запросы
- в API-слое надо валидировать id до запроса в БД

Пример:

```js
new ObjectId(id)
```

нужно делать только после валидации формата.

## 23. Bulk operations часто лучше одиночных запросов
Если надо массово менять данные, стоит думать про:

- `bulkWrite`
- батчи
- ordered vs unordered execution

Это особенно полезно в:
- миграциях
- синхронизациях
- ingestion pipeline
- фоновых задачах

## 24. Нужны миграции данных, даже в MongoDB
Частая иллюзия: “схемы нет — миграции не нужны”.

На практике нужны:
- versioning документов
- backfill новых полей
- миграции индексов
- поэтапный rollout новой схемы
- backward compatibility на время перехода

Правильный подход:
- код должен переживать старую и новую версии документов
- миграция должна быть повторяемой
- большие миграции — батчами

## 25. Наблюдаемость обязательна
Сеньор должен следить не только за приложением, но и за БД.

Нужно мониторить:
- медленные запросы
- использование индексов
- размер коллекций и индексов
- latency
- connection pool saturation
- lock/contention симптомы
- replication lag
- cache hit / working set behavior

## 26. Безопасность — не “потом”
Важно помнить:

- не логировать чувствительные поля целиком
- не хранить секреты в открытом виде
- включать auth/tls
- ограничивать роли по принципу least privilege
- не прокидывать пользовательский input напрямую в Mongo-запрос без валидации

Особенно опасны:
- operator injection вроде `$ne`, `$gt` в фильтрах
- небезопасные динамические сортировки и проекции

## 27. Нужно проектировать под конкретные access patterns
Самая сеньорская мысль про MongoDB:

**сначала запросы и сценарии доступа, потом схема.**

Не:
- “давайте сделаем красиво по объектам домена”

А:
- какие 5–10 запросов самые горячие?
- какие SLA?
- где нужны атомарность и консистентность?
- где допустима денормализация?
- какие поля фильтруются, сортируются, агрегируются?

MongoDB особенно сильно наказывает за модель “сначала абстрактная красота, потом реальные запросы”.

3. **Индексы — часть дизайна, а не оптимизация потом**
